# BUDS MVP — Технический дизайн

**Дата:** 2026-04-22
**Версия системы:** BUDS v1.1
**Scope:** MVP — первый этап разработки

---

## Контекст

**Администратор BUDS** — система ИИ-агентов для автоматизации цветочного магазина на Яндекс Маркете.

Ключевая специфика бизнеса:

- ~900 SKU, ~20 видов сырья (10 видов цветов + 10 расходников)
- 0–15 заказов/день, пиковые всплески в праздники
- Работа по модели FBS
- Яндекс Маркет агрессивно управляет ценой витрины — скидки за счёт продавца могут достигать 60–70% (реальный пример: заказ на 4 546 ₽, скидка Яндекса −3 065 ₽, выручка к выплате — 1 ₽)
- Термопринтер подключён к локальному компьютеру флориста (в будущем — мини-ПК)

Стартовая точка: VPS-сервер, магазин работает, токены Маркет API не подключены, Google Sheets создаётся с нуля.

**Команда:**

- **Владелец** — полный доступ через Telegram, принимает решения
- **Флорист** — отдельный Telegram-аккаунт, получает только уведомления о новых заказах и таймерах сборки. В будущем флористов может быть несколько; на MVP — один.

---

## Архитектурный подход: Модульный монолит с шиной событий

Один deployable (Docker Compose), агенты — изолированные модули с чёткими границами. Общаются через внутреннюю шину событий на Redis pub/sub.

**Почему не микросервисы:** оверкил для 0–15 заказов/день, замедлит разработку MVP.
**Почему не монолит:** агенты с прямыми вызовами быстро запутываются при 4+ агентах.
**Модульный монолит:** просто деплоить, легко отлаживать, при росте нагрузки любой агент выносится в отдельный сервис без переписывания.

## Поток событий: новый заказ

```
Маркет → POST /webhooks → event_bus.publish("order.created")
                                    │
         ┌──────────────────────────┼──────────────────────┐
         ▼                          ▼                      ▼
   Order Agent               Flower Stock            Print Agent
   регистрирует заказ        резервирует сырьё       отправляет задание
   запускает таймер 60 мин   (−2 буфер на брак)      через WebSocket
   уведомляет владельца      пересчитывает витрину    на локальный принтер
   и флориста в Telegram     НЕМЕДЛЕННО

Print Client (локально) ← WebSocket push → печатает ярлык мгновенно
```

**Важно:** Pricing Agent при получении заказа не участвует. Цена зафиксирована в момент покупки. Pricing Agent работает только проактивно по расписанию.

## Правило доступного остатка

```
Доступно для продажи = физический остаток − резерв − 2 (буфер на брак)
```

Пример: 50 роз на складе, заказ на 30 → резерв 30, буфер 2, доступно 18.
Пересчёт витрины Маркета происходит **немедленно** в момент резервирования, не после сборки.
Списание сырья — только после получения статуса «Готов» от Маркета.

## Поток событий: жизненный цикл заказа

```
order.created   → резерв сырья (−2 буфер), пересчёт витрины, печать ярлыка,
                  уведомление владельцу и флористу
order.ready     → списание сырья, остановка таймера
order.cancelled → снятие резерва, пересчёт витрины
order.timeout   → снятие резерва, пересчёт витрины, фиксация просрочки
order.shipped   → обновление статуса
order.delivered → обновление статуса
```

## Order Agent: таймеры и сценарий «Готов»

| Момент                                      | Действие                                                                                                                            |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Новый заказ                             | Уведомление**владельцу и флористу** в Telegram                                                          |
| T+45 мин (−15 мин)                         | «Заказ #123, осталось 15 минут» → владельцу и флористу                                               |
| T+50 мин (−10 мин)                         | «Осталось 10 минут» + две кнопки:**«Готов сейчас»** / **«Авто через 5 мин»** |
| Выбор «Авто» → T+55                   | Система отправляет статус «Готов» в Маркет API                                                         |
| Ожидание подтверждения       | 120 секунд на получение подтверждения от Маркета                                                     |
| Нет подтверждения за 120 сек | 🚨 «Статус не подтверждён! ~180 сек — зайдите в Маркет вручную»                               |
| T+60 мин без ready                          | `order.timeout` — резерв снят, витрина пересчитана                                                           |

**Кнопки «Готов сейчас» и «Авто через 5 мин»** — интерактивные кнопки в Telegram (inline keyboard aiogram).

## Pricing Agent: логика мониторинга цен

### Три уровня цен

1. **Цена каталога** — выставляется продавцом в кабинете Маркета
2. **Цена витрины** — что видит покупатель (после скидок и акций Яндекса)
3. **Минимальная цена реализации** — из Google Sheets

**Ключевой принцип:** цена каталога устанавливается с запасом, чтобы даже после всех вычетов Яндекса итоговая «Выручка к выплате» не падала ниже минимальной цены реализации.

### Структура удержаний (из реальных отчётов)

| Статья                                       | Описание                                                                         |
| -------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Вознаграждение по продажам | Базовая комиссия Маркета                                           |
| Стоимость доставки                | Доставка покупателю                                                    |
| **Скидка покупателю**        | ⚠️ Главный риск: скидка Яндекса за счёт продавца |
| Буст продаж                              | Расходы на продвижение                                               |
| Программа лояльности            | Бонусы покупателям                                                      |
| **Выручка к выплате**         | Итоговая сумма на руки                                                |

### Формулы ценообразования (Google Sheets → Маркет)

| Поле Маркета          | Формула                                | Описание                                                  |
| -------------------------------- | --------------------------------------------- | ----------------------------------------------------------------- |
| Цена                         | = себестоимость + наценка | Рассчитывается из стоимости сырья   |
| Зачёркнутая цена  | = Цена × 1.40                            | Фиксированная константа для всех SKU |
| Минимум для акций | = Цена × 1.10                            | Фиксированная константа для всех SKU |

### Расписание и логика сверки

**Расписание:** каждые 3 часа (APScheduler).

| Ситуация                                                      | Действие                                                                                                     |
| --------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Витрина < минимальной цены реализации | 🚨 Срочный алерт: «Продажа ниже порога, поднять каталожную цену» |
| Витрина < оптимальной цены                      | ⚠️ Алерт с предложением скорректировать                                           |
| Каталог расходится с Sheets                         | ⚠️ «Обновить цену в кабинете?»                                                              |
| Всё в норме                                                  | Ничего                                                                                                         |

При подтверждении владельцем → Pricing Agent обновляет каталожную цену через Маркет API.

### Карантин цен

Если новая цена сильно ниже текущей или рыночной — Маркет скрывает товар до ручного подтверждения. Pricing Agent предупреждает об этом риске перед отправкой изменения цены через API.

### Парсинг витрины

Читается через публичную выдачу Маркета (httpx + BeautifulSoup). Риск: антискрапинг. Альтернатива — если Маркет API предоставляет цену витрины, использовать его. Проверяется на этапе тестирования API.

### Фактические комиссии: автоматизация или ручная загрузка?

Яндекс Маркет API имеет эндпоинт `POST /campaigns/{id}/stats/orders` с финансовой статистикой. **На этапе тестирования API** проверяем: содержит ли он разбивку по статьям (включая «Скидку покупателю»). Если да — автоматизируем. Если нет — команда `/отчёт` в Telegram для ручной загрузки CSV.

Известные колонки из отчёта «Услуги и маржа по заказам»:
`order_id | статус | цена_товара | стоимость_услуг | выручка_к_выплате | вознаграждение_по_продажам | стоимость_доставки | буст_продаж | скидка_покупателю | программа_лояльности`

## Управление магазином (Оркестратор)

Команды через Telegram. Парсинг через LLM (Gemini 2.0 Flash).

**Команды закрытия:**

- `закрой магазин` → закрывает прямо сейчас
- `закрой на 2 часа` → закрывает + APScheduler открывает через 2 часа
- `закрой до 18:00` → закрывает + открытие запланировано на 18:00

**Команды открытия:**

- `открой магазин` → открывает по стандартному графику
- `открой завтра в 10:00` → APScheduler отправляет команду в **09:00** (за 2 час до желаемого времени, т.к. Маркет применяет изменения графика с задержкой ~2 час)

**Предупреждение:** при любом изменении графика система сообщает, что изменение вступит в силу через ~2 час.

**Механизм:** обновление графика через Маркет API. Доступность этого метода — **проверяется на этапе тестирования API**.

## Агенты MVP: границы ответственности

### Print Agent

**Триггер:** `order.created`**Действия:**

1. Скачивает URL ярлыка через Маркет API
2. Создаёт `print_jobs` запись со статусом `pending`
3. Отправляет задание через WebSocket на `print_client.py` → статус `sent`
4. Получает подтверждение печати → статус `done`
5. При ошибке или разрыве WebSocket → статус `failed` → алерт владельцу

**Граница:** только шильдик Маркета, ничего больше.

---

### Flower Stock Agent

**Триггеры:**

- `order.created` → резерв по рецепту (с учётом буфера −2) → **немедленно** обновить витрину Маркета
- `order.ready` → списать сырьё → зафиксировать себестоимость
- `order.cancelled` / `order.timeout` → снять резерв → обновить витрину
- Telegram: `«пришло 30 роз 40см по 80₽»` → приход, пересчёт средней себестоимости
- Telegram: `«списал 5 хризантем»` → порча
- Telegram: `«дополнительно списал 3 розы к заказу #123»` → extra_debit, корректировка себестоимости

**Пересчёт витрины:** математически проверить все 900 рецептов, обновить стоки через Маркет API.

**Начальная загрузка:** скрипт читает Google Sheets → загружает 900 SKU + рецепты + начальные цены.

---

### Order Agent

**Триггеры:** вебхуки Маркета
**Уведомления:** владелец + флорист(ы) — только о новых заказах и таймерах
**Таймеры:** T+45 мин, T+50 мин (кнопки), T+55 мин (авто-ready), T+60 мин (timeout)
**Подтверждение «Готов»:** 120 сек на ответ Маркета, затем алерт для ручного действия

### Pricing Agent

**Расписание:** каждые 3 часа
**Действия:** читает Sheets → читает кабинет → парсит витрину → сравнивает → алерты → при подтверждении обновляет цену
**Карантин:** предупреждает перед значительным снижением цены
**Экономика:** при доступности через API — автоматически; иначе команда `/отчёт`

### Оркестратор (управление магазином)

**Парсинг:** Gemini 2.0 Flash
**График:** команды с временными параметрами, APScheduler, учёт задержки Маркета ~1 час

## Что тестируем до написания кода (Этап 0)

Перед разработкой агентов необходимо проверить через Маркет API:

1. **Формат вебхуков** — какие поля приходят при каждом статусе заказа
2. **Финансовые данные** — есть ли в `stats/orders` разбивка по статьям (скидка, буст, комиссия)
3. **Управление стоками** — формат запроса на обновление остатков
4. **Управление ценами** — поведение карантина при изменении через API
5. **График работы** — есть ли API для изменения графика, задержка применения
6. **Ярлык заказа** — эндпоинт и формат ответа

## Порядок разработки

1. **Фундамент:** Docker Compose, PostgreSQL, Redis, FastAPI, aiogram, Alembic, токены Маркет API, тестирование API (см. выше)
2. **Print Agent:** WebSocket сервер, print_client.py, print_jobs
3. **Flower Stock Agent:** импорт из Sheets, учёт остатков с буфером −2, резервирование, пересчёт витрины
4. **Order Agent:** жизненный цикл, таймеры с кнопками, сценарий «Готов», уведомления флористу
5. **Pricing Agent:** Sheets API, сверка цен, карантин, алерты, импорт/API экономики
6. **Управление магазином:** LLM-парсинг, API графика, APScheduler с учётом задержки

---

## Критерии готовности MVP

- Новый заказ → ярлык напечатан мгновенно через WebSocket
- Витрина обновляется в момент резервирования, буфер −2 соблюдается
- Просрочка сборки: система пытается сама поставить «Готов», при неудаче — алерт
- Флорист и владелец получают уведомления о заказе и таймерах
- Цены мониторятся каждые 3 часа, критические алерты уходят немедленно
- Магазин открывается/закрывается одной командой с учётом задержки Маркета
- Дополнительное списание брака доступно через Telegram
