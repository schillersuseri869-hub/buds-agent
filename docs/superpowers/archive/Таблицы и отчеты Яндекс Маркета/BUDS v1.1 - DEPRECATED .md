# 🌸Администратор BUDS v1.1

## Система интеллектуальных агентов для управления цветочным магазином на Яндекс Маркете

---

## 📌 О системе

**Администратор BUDS** — это комплекс взаимосвязанных ИИ-агентов, автоматизирующих операционную деятельность цветочного магазина на Яндекс Маркете. Система спроектирована под специфику флористического бизнеса:

- Скоропортящийся товар (срок жизни цветов 3–7 дней)
- Виртуальные SKU, собираемые из общего пула сырья
- Малый объём заказов (0–15 в день) с пиковыми всплесками в праздники
- Эмоциональная природа покупки (подарки, праздники, поводы)
- Работа в формате FBS с перспективой перехода на DBS
- **Агрессивное управление ценой со стороны Яндекс Маркета** (скидки, акции, демпинг)
- **Высокая ценовая чувствительность клиентов** в сравнении с конкурентами

Система призвана **минимизировать ручной труд**, **гарантировать точность учёта**, **контролировать ценообразование**, **максимизировать маржу на пиках** и **обеспечить высокий уровень клиентского сервиса**.

## 🎯 Ключевые принципы архитектуры

### 1. Единая точка входа — Telegram

Владелец бизнеса взаимодействует с системой через один Telegram-бот, который маршрутизирует запросы между агентами.

### 2. Минимум отвлечения персонала

- **Флорист** работает только с интерфейсом Маркета, в нашу систему ничего не пишет (кроме редких нестандартных случаев).
- **Владелец** получает сводки, алерты и принимает решения — но не ведёт рутину.

### 3. Витрина Маркета управляется автоматически

Карточки на Маркете — это **виртуальные SKU**, собираемые из общего пула сырья. Система в реальном времени пересчитывает доступность и стоимость каждой карточки.

### 4. Прозрачность ценообразования через Google Sheets

Все расчёты цен ведутся в Google-таблице с понятными формулами. Владелец может в любой момент скорректировать формулы, коэффициенты, сценарии — не трогая код.

### 5. Точка истины — Яндекс Маркет

Статусы заказов берутся из вебхуков Маркета. Списание сырья привязано к моменту перехода заказа в статус «Готов к отгрузке».

### 6. Принцип экономии ресурсов

- Сложные задачи — дорогие LLM (GPT-4, Claude Sonnet)
- Рутинные задачи — дешёвые модели (GPT-4o-mini) или классический код
- При объёме 0–15 заказов/день система остаётся рентабельной

### 7. Постоянный контроль маржинальности

Ни один заказ не должен быть в убыток. Pricing Agent следит за ценой витрины в реальном времени и блокирует работу с карточкой при риске ухода в минус.

## 👥 Команда агентов

### 🖨 Агент 1: Print Agent — «Печатник»

**Роль:** Автоматическая печать шильдиков (ярлыков Маркета) на термопринтере.

**Что делает:**

- Получает вебхук о новом заказе
- Скачивает PDF-ярлык из Яндекс Маркета через API
- Отправляет на печать на термопринтер
- Уведомляет об успехе/ошибке

**Что НЕ делает:**

- Не печатает ничего, кроме шильдика Маркета
- Не создаёт карточек пожеланий, чеков, других документов

**Зависимости:**

- Получает команду от Оркестратора / вебхука Маркета
- Использует Маркет API для загрузки ярлыка
- Использует локальный принтер (подключение к серверу)

**Приоритет реализации:** 🟢 Высокий (быстрая победа, простая логика)

---

### 🌹 Агент 2: Flower Stock Agent — «Учёт цветов»

**Роль:** Центральный агент системы. Ведёт учёт сырья, управляет виртуальной витриной Маркета, считает себестоимость.

#### Ключевые концепции

**Двухуровневый учёт остатков:**

- **Физический остаток** — сколько реально лежит на складе
- **Резерв** — сырьё, заблокированное под активные заказы
- **Доступный остаток** = Физический − Резерв → это то, что видит Маркет

**Рецепты карточек:**
Для каждой карточки на Маркете описан состав (розы × N шт, упаковка × 1, лента × 1 м). Система математически связывает виртуальные SKU с реальным сырьём.

**Управление витриной:**
После любого изменения остатков система пересчитывает, какие карточки можно собрать, и обновляет стоки на Маркете через API.

#### Триггеры и реакции

| Событие                           | Физ. остаток | Резерв | Доступно |  Витрина  |
| ---------------------------------------- | :--------------------: | :----------: | :--------------: | :--------------: |
| Приход сырья                  |           +           |      0      |        +        | Пересчёт |
| Заказ «Ждёт сборки»     |           0           |      +      |        −        | Пересчёт |
| Отмена до «Готов»         |           0           |      −      |        +        | Пересчёт |
| Статус «Готов»              |           −           |      −      |        0        |  Без изм.  |
| Возврат после «Готов» |           0           |      0      |        0        |  Без изм.  |
| Списание порчи              |           −           |      0      |        −        | Пересчёт |
| Просрочка 60 мин             |           −           |      −      |        +        | Пересчёт |

#### Функции

1. **Приём сырья** — ручной ввод через Telegram («пришло 30 роз красных 40см по 80₽»)
2. **Автоматическое резервирование** при получении заказа
3. **Автоматическое списание** при переходе заказа в статус «Готов к отгрузке»
4. **Учёт расходников** (упаковка, ленты, оазис, коробки) — списываются вместе с цветами по рецепту
5. **Пересчёт витрины** на Маркете в реальном времени
6. **Напоминания о старой партии** — ежедневно, начиная с 7 дней хранения
7. **Списание порчи** — по команде («списал 8 роз, завяли»)
8. **Расчёт себестоимости** каждого собранного букета
9. **Передача себестоимости в Pricing Agent** для обновления формул в Google Sheets
10. **Переклассификация списаний** при возвратах (продажа → потеря)
11. **Отчёт по потерям** — сколько цветов ушло в списание, динамика

#### Исключение: индивидуальные букеты

Если заказ нестандартный (без фиксированного рецепта), флорист один раз сообщает состав в Telegram. Это единственная ситуация, где он взаимодействует с нашей системой.

**Приоритет реализации:** 🟢 Наивысший (фундамент всей системы)

### 📦 Агент 3: Order Agent — «Диспетчер заказов»

**Роль:** Управление жизненным циклом заказа от получения до доставки.

**Что делает:**

1. Получает вебхук о новом заказе от Маркета
2. Передаёт команду Flower Stock на резерв сырья
3. **Запрашивает у Pricing Agent проверку цены этого заказа** — не ушёл ли в убыток из-за скидки Маркета
4. Запускает таймер 60 минут на сборку букета
5. Отправляет команду Print Agent на печать шильдика
6. Отслеживает статусы заказа (Ждёт сборки → Готов → SHIPPED → DELIVERED)
7. Обрабатывает пограничные случаи (отмены, возвраты, просрочки)
8. Ведёт единый реестр активных заказов

**Таймеры и алерты:**

- **T+45 мин** после создания заказа: «осталось 15 минут на сборку»
- **T+55 мин**: экстренный алерт «5 минут!»
- **T+60 мин без "Готов"**: фиксация просрочки, снятие резерва

**Не проверяет наличие сырья** — наличие гарантируется самой витриной (если клиент смог купить, сырьё есть).

**Зависимости:**

- Получает данные от Маркета через вебхуки и API
- Командует Flower Stock Agent (резерв/списание/снятие резерва)
- Командует Print Agent (печать)
- Взаимодействует с Pricing Agent (проверка маржинальности)
- Уведомляет Владельца при нестандартных ситуациях

**Приоритет реализации:** 🟢 Высокий (нужен сразу после Flower Stock)

---

### 💰 Агент 4: Pricing Agent — «Контролёр цен» ⭐

**Роль:** Автоматическое управление ценообразованием: расчёт оптимальной цены, контроль реальной цены на витрине Маркета, мониторинг конкурентов, рекомендации по корректировке.

#### Почему этот агент критичен

Яндекс Маркет **агрессивно управляет ценой витрины**:

- Автоматически применяет скидки «за счёт продавца»
- Добавляет акции «Выгодная цена», «Скидка дня»
- Использует механизмы «Цена от N ₽» — давит к снижению
- Меняет цену витрины относительно вашей базовой

**Последствие без контроля:** вы поставили 3 500 ₽, а Маркет показывает 2 800 ₽ → минус маржа, иногда в убыток.

Кроме того, в цветочном бизнесе **цена конкурентов = важнейший фактор покупки**. Клиент видит выдачу Маркета, где рядом с вашими букетами стоят аналогичные от других флористов.

#### Функция 1: Google Sheets — «Калькулятор цен»

**Структура таблицы:**

| Столбец                              | Описание                                        |
| ------------------------------------------- | ------------------------------------------------------- |
| SKU карточки                        | Идентификатор                              |
| Название                            | Человекочитаемое имя                 |
| Состав                                | Рецепт из Flower Stock                          |
| Себестоимость                  | Автоматически из Flower Stock            |
| Желаемая маржа                 | % (ручная настройка)                     |
| Комиссия Маркета             | %                                                       |
| Доставка                            | ₽                                                      |
| Упаковка+прочее               | ₽                                                      |
| **Минимальная цена**   | Формула: точка безубыточности |
| **Оптимальная цена**   | Формула: с желаемой маржой        |
| **Максимальная цена** | Для пиков (+30–50%)                            |
| Текущий режим                   | Обычный / Акция / Пик                    |
| **Цена для Маркета**    | Итоговая цена продажи                |

**Двусторонняя синхронизация:**

- Flower Stock → Sheets: обновление себестоимости при изменении закупочных цен сырья
- Sheets → Pricing Agent: получение актуальной «Цены для Маркета»
- Pricing Agent → Маркет API: автоматическое обновление базовой цены

**Преимущество Google Sheets:**
Вы можете в любой момент скорректировать формулы, добавить коэффициенты, сезонные факторы — не трогая программный код. Максимальная гибкость.

#### Функция 2: Ежедневная сверка трёх цен

**Что сверяется:**

| Цена                                           | Источник                                                              |
| -------------------------------------------------- | ----------------------------------------------------------------------------- |
| **Оптимальная цена**          | Google Sheets (расчёт по формуле)                              |
| **Базовая цена продавца** | Кабинет Яндекс Маркета                                    |
| **Цена витрины**                  | Публичная выдача Маркета (с учётом скидок) |

**Частота проверок:**

- Обычные дни: каждые 3 часа (8 раз в день)
- Перед праздниками: 4 раза в день
- При изменении себестоимости: внеплановая проверка

**Классификация расхождений:**

| Ситуация                                | Действие                                                                 |
| ----------------------------------------------- | -------------------------------------------------------------------------------- |
| ✅ Всё сходится                      | Ничего                                                                     |
| ⚠️ Скидка Маркета                | Алерт: «Маркет снизил цену. Снять участие?»   |
| ⚠️ Цена в кабинете ≠ Sheets     | Алерт: «Цена в кабинете устарела. Обновить?» |
| 🚨 Продажа в убыток               | Срочный алерт: «Карточка уходит в минус!»      |
| 🚨 Цена ниже себестоимости | Критично: «Срочно скрыть карточку!»                |

#### Функция 3: Мониторинг конкурентов

**Что делает агент:**

1. **Реестр конкурентов** — список магазинов и ссылок на их карточки-аналоги
2. **Ежедневный парсинг** цен витрины конкурентов
3. **Сопоставление** своих и чужих карточек по категориям
4. **Формирование отчёта:**
   - Средняя цена рынка
   - Ваша позиция (выше/ниже среднего)
   - Рекомендации по корректировке
5. **Усиленный мониторинг перед праздниками** — отслеживание, когда конкуренты поднимают цены

**Пример отчёта:**
📊 Ценовой анализ на 15.11

✅ Букет 15 красных роз: ваша 3500, средняя 3400 (+3%)
⚠️ Пионы микс: ваша 4500, средняя 3900 (+15%) — снизить
🔥 Тюльпаны 25шт: ваша 2800, средняя 3200 (−12%) — можно поднять

#### Функция 4: Динамическое ценообразование

**Режимы работы:**

1. **Обычный** — цена из «Оптимальная»
2. **Акция** — временное снижение для буста продаж
3. **Предпраздничный** — плавное повышение за 7–10 дней до пика
4. **Пик** — максимальная цена (+30–50%)
5. **Послепраздничный** — возврат к обычным, утилизация остатков

**Автоматические сценарии:**

- **«Товар залежался»:** 3+ дня без продаж → предложение снизить на 10–15%
- **«Горячий товар»:** быстрее, чем обновляется остаток → поднять на 10%
- **«Конкурент повысил»:** 3+ конкурентов подняли → предложение следовать рынку
- **«Конкурент демпингует»:** не снижать, усилить маркетинг

#### Функция 5: История цен и эластичность

- Ваши цены по дням
- Цены конкурентов по дням
- Цены витрины Маркета (с учётом скидок)
- Связка «цена → количество заказов» для оценки эластичности

**Даёт ответы:**

- «Какая цена оптимальна по опыту 8 марта прошлого года?»
- «При какой цене падает спрос?»
- «Сколько потеряли, когда Маркет давал скидку?»

#### Взаимодействие с другими агентами

- **Flower Stock → Pricing:** актуальная себестоимость
- **Pricing → Маркет API:** чтение/запись базовых цен
- **Pricing → Marketing Agent:** сигналы о демпинге конкурентов
- **Pricing → Holiday Agent:** синхронизация предпраздничного роста цен
- **Pricing → Order Agent:** проверка маржинальности при получении заказа
- **Pricing → Analytics Agent:** данные для анализа эластичности

**Приоритет реализации:** 🟢 Наивысший (первая волна, наравне с Flower Stock)

### 🎉 Агент 5: Holiday Agent — «Календарь пиковых дат»

**Роль:** Управление подготовкой к праздникам — главному источнику выручки цветочного бизнеса.

**Ключевые даты:**

- 14 февраля — День влюблённых
- 8 марта — главный пик года
- 1 сентября — День знаний
- 25 мая — последний звонок
- 1 октября — День учителя
- Свадебный сезон (май–сентябрь)
- Новый год
- Персональные даты клиентов

**Что делает:**

1. **За 3 недели до пика:**

   - Напоминает запустить рекламу
   - Советует увеличить закупку сырья
   - Даёт прогноз на основе прошлых периодов
2. **За 7–10 дней до пика:**

   - Передаёт команду Marketing Agent на буст
   - **Передаёт команду Pricing Agent на плавное повышение цен**
   - Алерт: «проверь расходники — упаковку, ленты»
3. **За 2–3 дня до пика:**

   - Прогноз ожидаемого объёма заказов
   - Финальный чек-лист готовности
   - **Команда Pricing: переход в режим «Пик»**
4. **В день пика:**

   - Координирует работу, напоминает приоритеты
   - Мониторит скорость отработки заказов
5. **После пика:**

   - **Команда Pricing: возврат к обычным ценам**
   - Анализ результатов
   - Рекомендации к следующему году

**Взаимодействие с другими агентами:**

- Marketing Agent (запустить кампании)
- Pricing Agent (изменить режим ценообразования)
- Flower Stock (ожидаемый рост закупок)
- Analytics (прогнозы на основе истории)

#### Функция 4: Динамическое ценообразование

**Режимы работы:**

1. **Обычный** — цена из «Оптимальная»
2. **Акция** — временное снижение для буста продаж
3. **Предпраздничный** — плавное повышение за 7–10 дней до пика
4. **Пик** — максимальная цена (+30–50%)
5. **Послепраздничный** — возврат к обычным, утилизация остатков

**Автоматические сценарии:**

- **«Товар залежался»:** 3+ дня без продаж → предложение снизить на 10–15%
- **«Горячий товар»:** быстрее, чем обновляется остаток → поднять на 10%
- **«Конкурент повысил»:** 3+ конкурентов подняли → предложение следовать рынку
- **«Конкурент демпингует»:** не снижать, усилить маркетинг

#### Функция 5: История цен и эластичность

- Ваши цены по дням
- Цены конкурентов по дням
- Цены витрины Маркета (с учётом скидок)
- Связка «цена → количество заказов» для оценки эластичности

**Даёт ответы:**

- «Какая цена оптимальна по опыту 8 марта прошлого года?»
- «При какой цене падает спрос?»
- «Сколько потеряли, когда Маркет давал скидку?»

#### Взаимодействие с другими агентами

- **Flower Stock → Pricing:** актуальная себестоимость
- **Pricing → Маркет API:** чтение/запись базовых цен
- **Pricing → Marketing Agent:** сигналы о демпинге конкурентов
- **Pricing → Holiday Agent:** синхронизация предпраздничного роста цен
- **Pricing → Order Agent:** проверка маржинальности при получении заказа
- **Pricing → Analytics Agent:** данные для анализа эластичности

- **6 марта** — Pricing Agent: усиленный мониторинг конкурентов каждые 3 часа
- **7 марта** — прогноз ожидаемого объёма заказов, Pricing в режиме «Пик» (+30–50%)
- **8 марта** — режим пика: мониторинг в реальном времени
- **9 марта** — Pricing возврат к предпраздничным ценам
- **10 марта** — Pricing возврат к обычным, Analytics: итоговый отчёт

### Сценарий 5: Негативный отзыв

1. Клиент оставляет 2 звезды: «цветы завяли на следующий день»
2. Support Agent классифицирует: «жалоба на свежесть»
3. Готовит эмпатичный ответ с предложением компенсации
4. Эскалирует Владельцу: «согласовать компенсацию 30% или замену?»
5. После решения — публикует ответ под отзывом

### Сценарий 6: Конкурент демпингует

1. **09:00** — Pricing Agent парсит конкурентов
2. Обнаружено: «Ромашки в крафте» у конкурента 1 800 ₽ (наша 2 200 ₽)
3. Алерт владельцу с тремя опциями:

- Снизить цену (но не ниже минимальной 1 900 ₽)
- Не снижать, усилить рекламу → команда Marketing Agent
- Игнорировать (возможно разовая акция конкурента)

4. Pricing Agent 3 дня мониторит — если конкурент держит цену, повторный алерт

## 🔧 Технологический стек

### Инфраструктура

- **Сервер:** VPS (Ubuntu 22.04), 4 CPU / 8 GB RAM
- **База данных:** PostgreSQL 15
- **Кэш/очереди:** Redis
- **Контейнеризация:** Docker + Docker Compose
- **Мониторинг:** Uptime Kuma, Grafana (опционально)

### Код

- **Язык:** Python 3.11+
- **Фреймворк:** FastAPI
- **Telegram-бот:** aiogram 3.x
- **ORM:** SQLAlchemy 2.0
- **Очереди задач:** Celery + Redis
- **Планировщик:** APScheduler
- **Парсинг:** Playwright / httpx + BeautifulSoup

### LLM

- **Сложные задачи** (Support, анализ отзывов, рекомендации по ценам): Claude Sonnet 4 / GPT-4
- **Средние задачи** (Flower Stock, классификация): GPT-4o-mini
- **Простые задачи** (маршрутизация, сверка): классический код

### Интеграции

- **Яндекс Маркет API** (заказы, статусы, стоки, цены, ярлыки, отзывы)
- **Яндекс Директ API** (кампании, ставки)
- **Яндекс Метрика API** (аналитика трафика)
- **Google Sheets API v4** (калькулятор цен)
- **Telegram Bot API** (интерфейс)
- **ESC/POS** (драйвер термопринтера)

## 📦 Схема данных (высокоуровневая)

### Основные сущности

**1. Raw Materials (Сырьё)**

- Цветы по SKU (сорт, цвет, длина)
- Расходники
- Физический остаток, резерв, дата поставки, себестоимость

**2. Market Products (Карточки Маркета)**

- SKU карточки, рецепт, цена продажи, статус

**3. Recipes (Рецепты)**

- Связь «Карточка ↔ Сырьё» с количествами

**4. Orders (Заказы)**

- ID Маркета, статус, время, состав, резерв, списание, себестоимость, выручка, маржа

**5. Stock Movements (Движения сырья)**

- Приход, расход, списание, причина, связь с заказом

**6. Events Log (Журнал событий)**

- Все действия системы, вебхуки, команды владельца

**7. Holidays (Календарь)**

- Даты, статус подготовки, история

**8. Reviews (Отзывы)**

- Текст, оценка, ответ, классификация

**9. Price History (История цен)** ⭐ *НОВАЯ*

- SKU, дата, тип цены (Sheets/Кабинет/Витрина), значение
- Связь с заказами для анализа эластичности

**10. Competitors (Конкуренты)** ⭐ *НОВАЯ*

- Магазины, ссылки на карточки-аналоги
- Категории соответствия

**11. Competitor Prices (Цены конкурентов)** ⭐ *НОВАЯ*

- Конкурент, SKU-аналог, дата, цена, наличие

**12. Price Alerts (Ценовые алерты)** ⭐ *НОВАЯ*

- Тип (скидка Маркета, демпинг, убыток), статус, действие владельца
